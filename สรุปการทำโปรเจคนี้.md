# สรุปการทำโปรเจค BoxiFY E-commerce

## ภาพรวมโปรเจค

โปรเจค **BoxiFY** เป็นระบบ E-commerce แบบครบวงจร รองรับหลายบทบาท (ลูกค้า, ร้านค้า, ไรเดอร์, แอดมิน) มีระบบสั่งซื้อ-ชำระเงิน-จัดส่ง-คืนสินค้า และการจัดการร้านค้า/สินค้า/คูปอง/แบนเนอร์

---

## สถานะและเทคโนโลยีหลัก

| ส่วน | เทคโนโลยี |
|------|-----------|
| **Frontend** | React 18, React Router 6, Tailwind CSS, Axios, Socket.IO Client, React Toastify, Motion |
| **Backend** | Node.js, Express 5, Prisma ORM |
| **ฐานข้อมูล** | MySQL (schema ตาม ecom1 + Prisma) |
| **Auth** | JWT, Passport (Google, Facebook OAuth) |
| **Realtime** | Socket.IO |
| **อัปโหลดรูป** | Multer, Cloudinary (สินค้า/แบนเนอร์), เก็บไฟล์ local (หลักฐานคืนสินค้า/สลิป) |
| **QR ชำระเงิน** | promptpay-qr |
| **AI (ถามคำถามสินค้า)** | Google Generative AI / Groq (optional) |

## บทบาทผู้ใช้ (Roles)

| Role | คำอธิบาย | หน้าที่หลัก |
|------|----------|-------------|
| **user** | ลูกค้า | ดูสินค้า, สั่งซื้อ, ชำระเงิน, ติดตามออเดอร์, ขอคืนสินค้า, กระเป๋าเงิน (Wallet) |
| **seller** | ร้านค้า | จัดการร้าน/สินค้า/ออเดอร์, กระเป๋าร้าน, จัดการคำขอคืน + อนุมัติ/แจ้งข้อพิพาท |
| **courier** (rider) | ไรเดอร์ | รับงานจัดส่ง, รับงานรับของคืน, อัปเดตสถานะ (รับของ → นำจ่าย → ส่งสำเร็จ) |
| **admin** | แอดมิน | อนุมัติร้าน, จัดการผู้ใช้/สินค้า/หมวดหมู่/แบนเนอร์, คำสั่งซื้อ, คำขอคืน, ข้อพิพาท (Dispute), การชำระเงิน, Analytics |

## ฟีเจอร์หลักแยกตามโมดูล

### 1. สินค้าและร้านค้า

- **สินค้า:** CRUD, หมวดหมู่, รูปหลายรูป (Cloudinary), ตัวเลือก variant (สี/ไซส์), ราคาปกติ/ลดราคา
- **ร้านค้า:** สมัครร้าน (Seller Onboarding), แอดมินอนุมัติ/ระงับ, กระเป๋าร้าน (Store Wallet), รายงานยอดขาย
- **การค้นหา:** ค้นหาสินค้า, ฟิลเตอร์

### 2. ตะกร้าและคำสั่งซื้อ

- **ตะกร้า:** เพิ่ม/ลดจำนวน, เลือกตัวเลือกสินค้า, คำนวณรวม
- **สร้างออเดอร์:** ที่อยู่จัดส่ง, โค้ดส่วนลด (คูปอง), ค่าขนส่ง (preview shipping)
- **Flow พิเศษ:** กด "ซื้อสินค้า" จากหน้ารายละเอียด → เพิ่มลงตะกร้าแล้วไปหน้า cart พร้อมติ๊กถูกสินค้านั้นให้อัตโนมัติ

### 3. การชำระเงิน

- **วิธีชำระ:** เงินสด (COD), QR Code (PromptPay)
- **QR:** สร้าง payment → ได้ QR ให้ลูกค้าโอน ตามยอดออเดอร์ (ไม่มี Dynamic Amount แบบสุ่มเศษสตางค์)
- **อัปโหลดสลิป:** ลูกค้าอัปโหลดสลิป → สถานะรอแอดมินอนุมัติ (waiting_approval) → แอดมินอนุมัติ/ปฏิเสธ
- **SMS Webhook:** `POST /payments/webhook/sms` รับการแจ้งจาก Gateway/แอปดัก SMS — จับคู่ด้วย `transactionId` หรือ `orderId` ที่ส่งมา (ไม่จับคู่ด้วยยอดเงินจากข้อความ SMS)
- **Wallet ลูกค้า:** คืนเงินเมื่อยกเลิก/คืนสินค้า เก็บใน Wallet; จ่ายด้วย Wallet ได้

### 4. Logistics (การจัดส่ง)

- **สถานะออเดอร์:** PENDING → PENDING_CONFIRMATION (ชำระแล้ว) → PROCESSING (ร้านยืนยัน) → READY_FOR_PICKUP → RIDER_ASSIGNED → PICKED_UP → SHIPPED → OUT_FOR_DELIVERY → DELIVERED → COMPLETED
- **Shipment:** สร้างอัตโนมัติเมื่อร้านกด "พร้อมให้ไรเดอร์มารับ" (READY_FOR_PICKUP)
- **ไรเดอร์:** รับงาน (assign) → รับของ (IN_TRANSIT) → กำลังนำจ่าย (OUT_FOR_DELIVERY) → ส่งสำเร็จ (DELIVERED) พร้อมอัปเดต Order และ tracking_history
- **ลูกค้า:** ดูประวัติการติดตาม (TrackingTimeline), ยืนยันรับสินค้า (confirm received) → COMPLETED
- รายละเอียดเต็มใน **สรุปการทำขนส่ง.md**

### 5. การคืนสินค้า / คืนเงิน (Return & Refund)

- **รับประกัน 7 วัน:** ลูกค้าขอคืนได้เมื่อออเดอร์จัดส่งแล้ว และภายใน 7 วัน (จาก receivedAt/deliveredTime)
- **Flow คืนสินค้า:**
  1. ลูกค้าสร้างคำขอคืน (เหตุผล, รายการ, รูปหลักฐาน) → Order.refundStatus = REQUESTED
  2. ร้านกด **อนุมัติและเรียกไรเดอร์ไปรับของ** → refundStatus = WAITING_FOR_PICKUP, สร้าง/อัปเดต ReturnPickup (ตาราง return_pickup)
  3. ไรเดอร์รับงาน (assign) → ReturnPickup.courierId, order.refundStatus = WAITING_FOR_PICKUP
  4. ไรเดอร์รับของจากลูกค้า (pickup) → refundStatus = RETURN_IN_TRANSIT, ReturnPickup.status = PICKED_UP
  5. ไรเดอร์ส่งของคืนร้าน (complete + อัปโหลดรูปหลักฐาน) → refundStatus = RETURN_DELIVERED
  6. ร้านกด **อนุมัติและคืนเงิน** → คืนเงินเข้า Wallet ลูกค้า, หัก Store Wallet, คืนสต็อก; refundStatus = APPROVED
- **ข้อพิพาท (Dispute):** เมื่อสถานะ RETURN_DELIVERED ร้านสามารถกด **แจ้งปัญหา/ส่งเรื่องให้แอดมิน** (กรอกเหตุผล + อัปโหลดรูป) → refundStatus = IN_DISPUTE → แอดมินเข้า Dispute Management อนุมัติคืนเงินหรือปฏิเสธ
- **หน้าลูกค้า:** หน้า "คำขอคืนสินค้าของฉัน" (/returns) แสดงสถานะย่อยตาม flow (รอไรเดอร์มารับของ, ไรเดอร์รับงานแล้ว, ไรเดอร์กำลังนำของคืนร้าน ฯลฯ) โดยดึงจาก order.refundStatus และ returnPickups

### 6. แอดมิน (Admin Panel)

- **เมนูหลัก:** แดชบอร์ด (Analytics), ผู้ใช้งาน, อนุมัติร้านค้า, สินค้า, หมวดหมู่, แบนเนอร์, คำสั่งซื้อ, คำขอคืนสินค้า, **จัดการข้อพิพาท (Dispute)**, การชำระเงิน, ตั้งค่าเว็บไซต์
- **Dispute Management:** แสดงรายการที่ refundStatus = IN_DISPUTE; ปุ่ม อนุมัติคืนเงินให้ลูกค้า / ปฏิเสธการคืนเงิน (เรียก return-approve API)

### 7. ไรเดอร์ (Rider Dashboard)

- **งานจัดส่ง:** งานที่รอรับ (available-jobs) → รับงาน (assign) → รับของ → กำลังนำจ่าย → ส่งสำเร็จ (อัปโหลดรูปหลักฐานได้)
- **งานรับของคืน:** งานรับสินค้าคืน (return-jobs) → รับงาน (assign) → รับของจากลูกค้า (pickup) → ส่งของคืนร้าน (complete + อัปโหลดรูปหลักฐาน) — ใช้ field ชื่อ `proof` ในการอัปโหลดหลักฐาน

### 8. อื่นๆ

- **การแจ้งเตือน:** Socket.IO + Notification (payment, order status)
- **คูปอง:** สร้าง/ใช้โค้ดส่วนลด (validate ที่ checkout)
- **รีวิว:** ลูกค้าให้คะแนน/รีวิวสินค้า; ร้านดูรีวิวร้าน
- **AI ถามคำถามสินค้า:** ใช้ Gemini/Groq ตอบคำถามเกี่ยวกับสินค้า (optional)

## ฐานข้อมูลและสคริปต์ช่วยเหลือ

- **Prisma schema:** อยู่ที่ `server/prisma/schema.prisma` (User, Order, Shipment, Product, Payment, OrderReturn, ReturnPickup, Wallet, Store ฯลฯ)
- **refundStatus (Order):** ค่าได้แก่ NONE, PENDING, REQUESTED, WAITING_FOR_PICKUP, RETURN_IN_TRANSIT, RETURN_DELIVERED, **IN_DISPUTE**, APPROVED, REJECTED (ต้องมี ENUM ใน MySQL ครบ จึงจะอัปเดตได้)
- **ตาราง return_pickup:** ใช้เก็บงานไรเดอร์รับของคืน; ถ้าไม่มีใน DB จะสร้างผ่าน `add-missing-columns.js`
- **สคริปต์ `add-missing-columns.js`:** รันตอนสตาร์ท server — สร้างตาราง/คอลัมน์ที่ขาด (รวม SiteSetting, Banner, Notification, return_pickup) และอัปเดต ENUM refundStatus ให้ครบ

## การรันโปรเจค

- **Backend:** `cd server && npm start` (พอร์ตตาม env เช่น 3000)
- **Frontend:** `cd client && npm start` (มักใช้ proxy ไป backend, พอร์ตเช่น 3001)
- **DB:** ต้องมี MySQL และตั้ง `DATABASE_URL` ใน `.env`; ครั้งแรกอาจรัน Prisma migrate หรือให้ add-missing-columns สร้างสิ่งที่ขาด
